# 스펙 확인 게이트 상세 (spec-gate)

이 절차는 코어 "스펙 확인 게이트" 골격(3층 존재 · 🙋=의도 안전장치 · 게이트는 한눈에+🙋만 띄움 · LIGHT는 스위치 조건)의 **상세**다.

## 스펙 3층 (모든 스펙에 적용, 브레인스토밍→스펙 작성 시)

1. **한눈에 (≤10줄 · 조망 · 고정 슬롯):** 스펙 맨 위(제목 다음). **자유서술 금지, 정해진 칸을 채운다**(요약이 위험을 지워버리는 사고 방지). = **핵심5(현재형)** — 왜(목적) · 무엇 · 범위(안 하는 것 포함) · 잘되면(기대효과) · **⚠ 최대 위험**(핵심5 잘못되면의 **1개 투영** — 가장 큰 위험 하나, 필수·생략 불가). 칸 뜻·위험 작성법은 "공통 코어" 참조. 역할=조망. 흘려봐도 안전이 안 깨진다(안전은 🙋+뒤 게이트 담당). "큰 그림 어긋남"을 한 번에 감지하는 용도이지 안전장치가 아니다.
2. **🙋 확인 필요 (2~4개 · 안전장치):** 브레인스토밍에서 사용자가 **명시 안 해 AI가 대신 정한 결정(interpolation)** 만 나열. 이미 합의됐거나(예: 의식적으로 수용한 트레이드오프)·뒤 게이트가 잡을 기술 위험은 **넣지 않는다.** 각 항목은 **예/아니오로 결정 가능**하게("v1에서 X 빼고 v2로 미뤘어요 — 맞나요?"). 이건 문서가 아니라 **질문**이라 피로해도 답하게 된다(질문은 답하고 문서는 흘려본다 — 핵심 차이). 권고 3개·**최대 4개**, 넘치면 "브레인스토밍 합의가 덜 됐다"는 신호로 되짚는다. 없으면 "🙋 확인 필요: 없음" 명시. **판단이 애매하면(합의된 건지 내가 대신 정한 건지 불분명) 🙋에 넣는다**(fail-safe: 안 묻고 넘기기보다 묻는다 — "의심되면 FULL" 기조와 동일). **🙋 우선순위:** 후보가 상한(4개)을 넘거나 무엇부터 물을지 가려야 하면 **답이 스펙의 구조·범위를 바꿀 결정부터** 올린다(껍데기 문구·나중에 바꿔도 싼 것보다 먼저). **단 안전·권한·데이터 노출·삭제 방식 결정은 "나중에 바꿔도 싼 것"으로 강등하지 말고 구조·범위급으로 취급해 항상 최우선**(뒤 게이트가 멈춰도 🙋에서 안 물으면 사용자가 미확인 통과하므로).
3. **본문 (완결 · 자산):** 지금처럼 완결되게(살아있는 지도·자산 보존). 게이트에서 **필수로 읽으라 하지 않는다** — 나중에 "왜 이렇게 정했지"를 되짚는 참조용.

## 게이트 동작

- "스펙 파일 읽어보세요"가 **아니라** `한눈에` + `🙋 확인 필요`만 대화창에 띄우고 🙋 항목을 결정받는다.
- **전체 스펙 파일 링크(경로)를 항상 함께 건다** — 강제 읽기가 아니라 "원하면 읽기"(자산 접근성).
- **LIGHT 자동 진행 조건은 "작업 규모 스위치"의 LIGHT 조건을 그대로 따른다**(별도 목록을 새로 적지 않는다 — 단일 출처). 그 절 3조건 중 **되돌리기·민감면 두 축**은 그대로 확인하고, **3번째("사용자가 할 일 구체적 — 모호한 새 기능 아님")는 여기선 🙋="없음"으로 확인한다** — 모호한 새 기능이면 AI가 대신 정한 결정(interpolation)이 생겨 🙋가 비지 않으므로 🙋="없음"이 곧 그 조건이다(따라서 3조건 모두 반영). 즉 되돌리기·민감면이 참이고 🙋도 "없음"이면 → **한눈에만 통보하고 다음 단계로 진행**(멈추지 않음). (🙋를 잘못 비운 경우의 최종 안전망은 뒤 plan-validator — 아래 항목대로 자동 진행이 후속 게이트를 건너뛰지 않는다.) "민감면"은 그 절 정의를 그대로 쓴다 — 여기서 재정의하지 않는다.
- **"자동 진행"은 plan-validator·후속 게이트를 건너뛰는 게 아니다** — 다음 단계(계획 수립 등)로 넘어가되 그 게이트들은 **그대로 거친다**. 스펙 확인이 가벼워졌다고 자동 검증 단계가 사라지지 않는다.
- 🙋에 항목이 있거나 민감면을 건드리면 → 그 항목만 확인받고 진행(전문 읽기 강제 아님).

이 게이트는 plan-validator(별도 심판)를 **대체하지 않는다** — 사람 확인(의도)과 자동 게이트(위험)는 보완이다.
