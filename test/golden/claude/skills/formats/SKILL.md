# 핵심5 — 세 양식(카드·한눈에·요약)이 공유하는 칸

**작업 시작 카드 · 한눈에 · 비전문가 요약은 같은 칸을 시점만 바꿔 쓴다.** 칸의 뜻·작성법(특히 위험)은 여기가 단일 원본 — 규칙을 바꿀 땐 여기만 고쳐 세 곳이 표류하지 않게 한다(중복 유지보수 제거). 실제 중복이 큰 건 **한눈에↔비전문가 요약**(거의 같은 칸을 현재/과거형으로).

**핵심5 (칸 정의):**
1. **왜(목적)** — 왜 이걸 하나(상위 이유·해결할 문제).
2. **무엇** — 뭘 만드나/했나(전문용어 없이 1~2문장).
3. **범위** — 하는 것 / **안 하는 것**.
4. **잘되면** — 잘되면 뭐가 좋아지나(기대효과).
5. **⚠ 잘못되면(위험)** — 발견된 위험을 **심각도 높은 순**으로(치명적인 게 사소한 것에 안 묻히게 · 신호 우선). 게이트 등급(blocker/high) 있으면 그 라벨 그대로. **생략 불가.** *두 투영(규칙은 같음):* 한눈에=가장 큰 **1개**(⚠최대위험·조망), 비전문가 요약=**빠짐없이 전부**(데이터 손실·보안·비용·노출 등 여러 개면 다 · 보고).

**시제:** 카드=하려는 것(미래) · 한눈에=만드는 것(현재·스펙) · 요약=한 것(과거).
**LIGHT:** 핵심5를 **한 줄로 압축**(무엇 + 발견 위험; 없으면 "위험 없음"). *— 비전문가 요약·한눈에에 적용. **작업 시작 카드의 LIGHT 한 줄은 "목표 + 성공 기준"** 이다(성공 기준은 LIGHT여도 생략 불가).*

# 비전문가 요약 (완료 후 · 보고 — 핵심5 과거형)

코드·spec·plan·아키텍처 결정 등 기술 산출물이 든 응답에 첨부한다(짧은 단순 질문·순수 대화엔 생략). 아래는 FULL 형식이며, LIGHT면 위 한 줄로 대체.

> **비전문가 요약** (핵심5 과거형 + 다음확인)
> - **지금 무엇을 했는가** (핵심5: 무엇)
> - **왜 이렇게 결정했는가** (핵심5: 왜 — 다른 선택지가 있었다면 무엇·왜 이걸)
> - **잘되면 어떻게 되는가** (핵심5: 잘되면)
> - **⚠ 잘못되면 어떤 위험인가** (핵심5: 잘못되면 — 요약은 **전부·심각도순**)
> - **다음에 확인할 것** (개발자에게 요청하거나 직접 테스트할 1~2개) — *요약 고유 칸.*

# 작업 시작 카드 (착수 전 · 정렬 — 핵심5 미래형 일부 + 고유 칸)

기술 작업 착수 전 먼저 제시한다(짧은 질문·순수 대화, 파일 1개·15분 이하 단순 작업엔 생략 — 진행 중 범위가 커지면 소급 제시). LIGHT면 한 줄 통보 후 곧장 진행.

> 📋 **작업 시작 카드**
> • 목표: 무엇을 이루려는가 (일상어 한 문장) — 핵심5: 무엇.
> • 범위/제약: 건드리는 것 / 절대 안 건드리는 것 — 핵심5: 범위.
> • 성공 기준: 체크 가능한 항목 2~5개 (모두 충족 = 성공). **각 항목에 "무엇으로(어떤 증거로) 충족을 잴지"를 함께 적는다(FULL 카드 한정).** 끝 점검이 그 증거로 채점하므로, 시작에 잣대를 못 박아 끝의 애매한 "된 것 같음"을 막는다. *(카드 고유 — "끝 점검" 채점 기준으로 재사용.)*
> • 길의 종류: [정해진 길=고정 절차] 또는 [탐색 필요=유연한 계획] — *카드 고유.*
> • 멈춤 규칙: 코어 "멈춤 규칙" 적용 — *카드 고유.*
> (왜·잘되면·잘못되면은 스펙 있는 새 기능이면 "한눈에"가 담당하므로 카드에선 생략. 착수 전 위험이 크면 ⚠잘못되면 한 줄만 덧붙인다.)

(제품지도 있으면) 이 작업이 feature-spec 어느 기능(F-○○)·IA 어느 화면인지 카드에 명시하고, 성공 기준을 그 기능 정의·관련 화면과 정합되게 작성. 성공 기준은 사용자와 정렬하며 "끝 점검" 채점 기준으로 재사용. 카드는 착수 전 "정렬"용, 비전문가 요약은 완료 후 "보고"용 — 시점이 달라 한 응답에 동시 출력하지 않는다.
