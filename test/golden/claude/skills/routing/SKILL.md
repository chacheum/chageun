# 모델·실행 라우팅 (상세)

코어 골격(안전 tie-break·게이트=최상위 모델·완료 보고≠검증)은 운영 규칙이 단일 원본이다 — 여기는 라우팅 표와 위임 세칙.

## 상시 규칙

서브에이전트를 띄울 땐 모델을 반드시 명시한다(미명시 시 세션 모델 상속(보통 최상위) → 병렬도 느리고 비쌈). 검증 게이트(plan-validator/pr-reviewer)·계획·스펙·아키텍처·복잡한 판단·최종 리뷰는 **무조건 최상위 추론모델**(현재 Opus — 메인 세션보다 약한 모델 금지). 명확한 기계적 구현(1~2 파일, 받아쓰기 수준)은 `code-implementer`(**Sonnet** 고정). 모델 등급과 실행 방식(인라인/서브에이전트/병렬)은 별개 축 — 아래 표로 조합해 라우팅한다.

**추론 강도(effort):** 게이트는 effort 미고정(세션 강도 상속 — 메인보다 안 약해지게). 절감은 일꾼만: `code-implementer`=`effort: medium`(뒤를 pr-reviewer가 받침). 메인 세션 effort는 차근이 못 건드림.

## GO 후 자동 결정

plan-validator GO(또는 사용자 "진행해도 좋다") 다음, 선택지를 띄우지 않고 **plan 작업 목록을 근거로 모델·실행 방식을 자동 결정해 한 줄로 통보한다**(+ "다르게 할까요?" 한 줄 override). 판정 인자: 작업 수 · 파일 겹침 · 의존성 · 보안/판단 여부 · 규모.

| 상황 | 모델·실행 |
| --- | --- |
| 보안·판단·권한·동시성·아키텍처·애매·복잡 | **Opus 인라인** (안전 바닥 — 절대 Sonnet 아님) |
| 명확·기계적(보안·판단 무관) · 2개+ 독립·파일 분리 | **Sonnet 서브에이전트 병렬** (아래 '병렬 위임' 규칙 — SDD는 태스크별 순차라 병렬 구현엔 안 씀) |
| 명확·기계적 · 단일 · 1~2파일 안의 반복·분량(경계 지어짐) | **Sonnet 서브에이전트 단일** (`code-implementer`; 스폰 비용보다 이득일 때; 파일 많으면 병렬 2개+ 또는 Opus) |
| 여러 파일 통합·디버깅 | **Sonnet** (애매하면 Opus) |
| 단일 · 자잘(한두 줄) | **인라인 즉시** (스폰 오버헤드 회피, 세션 모델) |
| 독립성·명확성·안전성이 애매 | **인라인 Opus** (fail-safe: 안전하게 비싼 쪽) |

**코어의 안전 tie-break가 이 표보다 항상 우선한다** — 보안·판단·권한·동시성·아키텍처가 조금이라도 걸리면 규모·명확성·반복성 무관 무조건 Opus 인라인.

## 병렬 위임

요청 없어도 독립적(서로 다른 파일) 기계적 작업이 2개 이상이면 기본으로 `code-implementer`(Sonnet) 여러 개를 한 응답에 동시에 띄운다. 단:
- 같은 파일 동시 수정 금지(순차, 또는 `using-git-worktrees`로 worktree 격리 — 즉흥 생성 말고 스킬 절차로).
- **공유·등록류 파일**(라우터 등록·index/barrel export·공통 타입·DB 마이그레이션)은 병렬의 숨은 충돌점 → 병렬에서 빼고 메인이 병렬 완료 후 순차 통합.
- 의존성 있거나(B가 A 결과 필요) 보안·권한·동시성처럼 판단 중요한 코드는 병렬 금지(순차 또는 Opus 직접).
- 독립성·명확성이 애매하면 위임하지 말고 Opus가 직접 구현한다("애매하면 Opus", 한 줄 알림).
- **완료 보고 ≠ 검증:** 서브에이전트가 "다 됐다"고 해도 그대로 믿지 않는다 — 위임한 변경은 실제 바뀐 파일(diff)을 확인한 뒤에만 완료로 치고 다음 작업을 쌓는다(게이트 전·게이트 없는 변경에서 거짓 완료 차단).

`subagent-driven-development`(태스크별 순차 plan 실행) 능력은 통보 한 줄로 노출되어 사라지지 않는다 — 위 '병렬 위임'과 별개다(SDD는 병렬 구현 아님).
