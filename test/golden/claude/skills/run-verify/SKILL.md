# 실제 구동 검증 상세 (run-verify)

> **안전-치명은 코어 골격에 있다:** 운영 쓰기 검증 hard block(사용자 동의로도 우회 불가) · 격리 테스트환경 필수 · 증거 없인 ✅ 금지(마지막 수정 이후 새로 돌린 증거만). 이 규칙들은 항상 유효하며, 여기선 그 절차의 스텝 세부(어떻게 돌려보나)만 편다.

## 스텝

1. **띄워 본다(시각):** 화면을 띄워 관측(스크린샷, 필요 시 모바일/데스크톱). 이때 `design-system` 규칙과 대조 — 어긋난 곳 보고, 규칙에 없는 새 패턴이면 등록 제안(3택: 기존 규칙에 맞추기/새 규칙 등록/일회성 예외). 규칙 문서 없으면 대조 생략·알림.
2. **끝까지 써본다(동작):** 대표 + 엣지 데이터(빈값·잘못된 형식·경계값·아주 긴 값·중복)로 버튼·저장·삭제를 실제로 누른다. 성공 기준을 시나리오로 삼아 항목별 통과/실패 판정. 권한·역할·로그인 상태가 다르면 그 경우도.
3. **테스트 환경(코어 골격 참조):** 동작 검증은 반드시 격리된 테스트 환경에서. 운영 쓰기 검증 hard block·격리 필수 조건은 코어 골격을 따른다(여기서 재정의하지 않음).
4. **안 되면 디버깅(추측 금지):** `systematic-debugging`으로 재현→원인→수정→재구동, 통과까지 반복.
5. **보고:** 어떤 시나리오를 어떤 데이터로 돌려 무엇이 되고 안 됐는지 비전문가가 알아보게 정리(스크린샷).

## 도구 (플랫폼별)

- **Claude:** 띄우기·눌러보기는 `run`·`verify` 스킬 또는 Playwright.
- **Codex:** `run`/`verify` 대신 shell로 dev 서버 기동(백그라운드) → Playwright(있으면)로 화면·동작, 없으면 `curl`로 응답·상태코드 확인하고 "시각 확인 제한됨" 명시. Playwright·dev 기동 불가 환경이면 "동작 검증 안 됨 — 미검증 출시 위험"으로 보고만.
